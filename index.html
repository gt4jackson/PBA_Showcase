About this project

What is this?

PBA Showcase is an interactive demonstration of performance-based assessment tasks. Each task represents a different category from the PBA Classification System (v1.2), a taxonomy of 7 primary task categories and 6 secondary dimensions for designing and evaluating performance assessments.

Unlike multiple-choice tests, these tasks require you to do something — have a conversation, investigate data, write an analysis, or use tools to solve a problem. The system collects rich behavioral evidence about your process, not just your final answer.

Design constraints

ConstraintHow it's met
ScalableAll task content lives in swappable JSON scenario files. New variants take hours, not weeks.
Cost-efficientRuns in any browser with no backend. Scoring is automated.
Evidence-richEvery action, timing, and decision is logged — not just final outputs.
Positive UXWorkplace-realistic framing. Feels relevant, not like a sterile exam.
Time-boundedEach task completes in 10-20 minutes.

Who is this for?

ETS stakeholders exploring AI-augmented assessment, external partners evaluating PBA approaches, conference attendees seeing the taxonomy in action, and assessment designers learning how to classify and create performance tasks.

6 secondary dimensions

Every task is also classified along 6 secondary dimensions that describe how it's administered and scored.

Shell + construct pack design

Every task is built as two distinct layers: a task shell (the interaction mechanics — chat UI, workspace, data explorer, writing editor) and a construct pack (the domain content — questions, scenarios, datasets, rubrics).

TASK SHELL (fixed code) ├── Interaction mechanics ├── Evidence logger ├── UI components ├── Scoring engine └── Feedback renderer CONSTRUCT PACK (swappable JSON) ├── construct_id + name ├── dimensions[] ├── content (questions, data, sources) ├── scoring_rubric ├── feedback_templates └── persona / framing

The shell never references a specific construct. It reads dimension names, scoring criteria, persona details, and feedback templates from the construct pack at runtime.

Evidence collection

Every task emits events to a universal evidence logger that captures actions, timing, decisions, and revisions. Event types include: task_start/end, item_view, user_input, interaction, revision, and submission.

The construct is always transparent to the user. Before, during, and after each task, the system shows what's being measured and why — a core fairness principle (Messick, 1994).

Construct Pack format

Every task shell reads its content from a construct pack — a single JSON object that defines what's assessed, how questions are framed, and how responses are scored. Swapping the pack pivots the task to a new construct without touching the shell code.

Key design principles

PrincipleHow the JSON enforces it
Construct transparencyconstruct_name and dimensions[] are shown to the user before, during, and after the task
Scorer portabilityEach question carries its own scorer spec — the shell engine is generic
Persona flexibilityThe persona object sets name, avatar, role, and tone — no hardcoded characters
Dimension independenceDimensions are an array — add 3 or 6, the shell adapts automatically
Feedback templates{dimension} placeholder is filled at runtime from the dimension name
